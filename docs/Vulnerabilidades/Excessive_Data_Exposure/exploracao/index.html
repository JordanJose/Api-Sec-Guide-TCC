Exploração

Tendo em mente os questionamentos levantados ao final da sessão anterior, é possível realizar os testes de segurança a fim de descobrir onde a vulnerabilidade pode ser explorada.

A melhor forma de explorar essa vulnerabilidade é estudando o comportamento da API, lendo a sua documentação e analisando as respostas de todos os endpoints, procurando por informações em excesso que são enviadas para o cliente, por esse motivo que é importante o uso de um proxy.

Rota de debug​

Uma das formas para descobrir se uma API possui esse tipo de vulnerabilidade é consultar as rotas de debug, mesmo que não constem na documentação (será necessário um esforço para deduzir as possíveis rotas).

Rotas de debug são comumente usadas por desenvolvedores para realizar testes na aplicação no ambiente de desenvolvimento e, por essa razão, costumam trazer informações sensíveis de muitos usuários ou sistemas internos.

Porém, há casos de rotas de debug da aplicação que vão para o ambiente de produção e, até mesmo, acabam sendo documentadas, expondo informações sensíveis para qualquer um que conheça esse endpoint.

Ao consultar a documentação da VAmPI é possível encontrar um endpoit de debug, o /users/v1/_debug, que possui a seguinte descrição:

Trecho de documentação da VAmPI
...
description: Displays all details for all users
...

Traduzindo, significa que essa rota retornará todos os detalhes de todos os usuários da aplicação. Podendo ser uma vulnerabilidade muito grave, caso não haja nenhum tipo de controle de acessos.

Então, para consultar se há a exposição excessiva de dados, basta realizar uma requisição do tipo GET para a rota /users/v1/_debug, já que não há a necessidade de informar nenhum tipo de token de autenticação.

Request para o endpoint de debud da aplicação
GET /users/v1/_debug HTTP/1.1
User-Agent: PostmanRuntime/7.29.2
Accept: */*
Postman-Token: 9b2c1f71-db86-424e-b161-4a4bef1cad51
Host: localhost:5000
Accept-Encoding: gzip, deflate
Connection: close
